## PART 1: Shared Data Model

### Config File Format: `config/agents.json`

This file MUST exist before the server starts. It defines all valid aliases.

```json
{
  "agents": [
    {
      "alias": "pta",
      "description": "Portfolio Trading Agent",
      "is_human": false
    },
    {
      "alias": "scanner",
      "description": "Stock Scanner Agent",
      "is_human": false
    },
    {
      "alias": "human",
      "description": "Human CLI Interface",
      "is_human": true
    }
  ]
}
```

Rules:
- `alias` must be unique and non-empty.
- `is_human` marks human interfaces. There can be multiple human aliases.
- Only aliases listed here are allowed to register. All others are rejected.

---

### Wire Protocol: JSON Message Format

Every message on the TCP wire looks exactly like this:

```json
{
  "from": "pta",
  "to": "human",
  "msg_type": "data",
  "timestamp": 1713100000.123,
  "byte_count": 42,
  "data": { "key": "value" }
}
```

Field definitions:
- `from` (string, required): Sender alias. Must match a registered agent.
- `to` (string, required): Recipient alias. Must match a registered agent or `"human"` for broadcast.
- `msg_type` (string, required): One of: `"registration"`, `"data"`, `"error"`, `"ack"`.
- `timestamp` (float, required): Unix epoch timestamp with millisecond precision.
- `byte_count` (integer, required): Length in bytes of the JSON-serialized `data` field.
- `data` (object, required): The payload. Content depends on `msg_type`.

#### msg_type = "registration"

Sent by an agent to the router when it first connects.

```json
{
  "from": "pta",
  "to": "router",
  "msg_type": "registration",
  "timestamp": 1713100000.0,
  "byte_count": 27,
  "data": { "alias": "pta" }
}
```

#### msg_type = "data"

A normal message between agents.

```json
{
  "from": "pta",
  "to": "human",
  "msg_type": "data",
  "timestamp": 1713100001.0,
  "byte_count": 85,
  "data": { "symbol": "AAPL", "price": 185.23, "signal": "BUY" }
}
```

#### msg_type = "error"

Generated by the router (from = "system") to inform human users about problems.

```json
{
  "from": "system",
  "to": "human",
  "msg_type": "error",
  "timestamp": 1713100002.0,
  "byte_count": 52,
  "data": { "error": "Agent 'scanner' timed out" }
}
```

#### msg_type = "ack"

Read receipt. Sent by the recipient back to the original sender to confirm message receipt.

```json
{
  "from": "human",
  "to": "pta",
  "msg_type": "ack",
  "timestamp": 1713100003.0,
  "byte_count": 45,
  "data": { "ack_for": 1713100001.0 }
}
```

The `ack_for` field contains the `timestamp` of the original message being acknowledged.

**Edge Cases:**
- Client disconnects without deregistering → cleanup in finally-block of _handle_client
- First message is NOT registration → close connection immediately

- Human client sends wrong format → return error + echo
- Human client sends pure alias at start → accept as login
